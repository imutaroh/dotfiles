# タスクダッシュボード（task-dashboard）

個人の「やりたいこと・タスク」を markdown ファイルで管理し、claude.ai の Artifact（固定URL・プライベート）にダッシュボードとして公開・更新し続けるスキル。

```
ユーザーの指示 → メインが指示を具体化 → 【Sonnetサブエージェント: tasks.md更新→HTML更新→同URLに再公開】 → メインが結果を報告
```

更新作業は定型なので、メインセッションのトークンを使わず **Sonnet のサブエージェントに委譲する**のがこのスキルの基本形。メインは「指示の具体化」と「結果の報告」だけを行う。

## 大前提（この構成が崩れると破綻する）

- **真実のデータは tasks.md。Artifact は表示専用のビュー。** Artifact はバックエンドのない静的ページなので、ブラウザ上の操作は保存されない。タスクの変更は必ずこのスキル経由（＝ファイル更新→再公開）で行う。
- **ダッシュボードは1つだけ。** 毎回同じファイルパス（dashboard.html）から同じURLへ再公開する。新しいURLを勝手に作らない。
- タスクファイルはユーザーが手でエディタから編集してもよい。パースできない行があっても壊さず、そのまま「その他」として表示する。

## ファイル

| ファイル | 役割 |
| --- | --- |
| `~/.claude/task-dashboard/tasks.md` | タスクの実体（真実のデータ）。実体は dotfiles の `.claude/task-dashboard/tasks.md` |
| `~/.claude/task-dashboard/dashboard.html` | 生成するダッシュボードHTML（毎回上書き） |

git コミットはしない（tasks.md は頻繁に変わる個人データ。コミットするかはユーザーの判断に任せる）。

## tasks.md フォーマット

```markdown
---
artifact_url: https://claude.ai/code/artifact/xxxx（初回公開後に記録）
updated: 2026-07-09
---

# タスクボード

## カテゴリ名（例: 学習 / 開発 / 生活）

- [ ] 未着手タスク | 優先度:高 | 期限:2026-07-15 | メモ:補足
- [~] 進行中タスク | 追加:2026-07-01
- [!] ブロック中タスク | メモ:何を待っているか
- [x] 完了タスク | 完了:2026-07-08
```

- ステータス記号: `[ ]`=未着手 / `[~]`=進行中 / `[!]`=ブロック / `[x]`=完了
- メタ情報は `| キー:値` を後ろに足す方式。全部省略可。優先度は 高/中/低。
- 日付が必要なとき（追加日・完了日・updated）は必ず `date +%Y-%m-%d` で取得する。捏造しない。
- カテゴリはユーザーの言葉に合わせて柔軟に作る。指定がなければ既存カテゴリに寄せ、合うものがなければ「その他」。

## 手順（メインセッションの動き）

### 1. 指示を具体化する

tasks.md を読み、ユーザーの指示を「どの行に何をするか」まで具体化する。

- 曖昧な指示（どのタスクか特定できない、カテゴリ不明等）は、サブエージェントに投げる**前に** AskUserQuestion で確認する。サブエージェントはユーザーに質問できない
- **表示だけの依頼（「進捗どう？」等）はサブエージェント不要。** メインが tasks.md を読んでサマリーと `artifact_url` を答えて終わり（ファイルも Artifact も変更しない）
- 初回（tasks.md が無い）は、frontmatter（artifact_url は空）と `# タスクボード` の雛形＋ユーザーが挙げたタスクを、これもサブエージェントに作らせてよい

### 2. Sonnet サブエージェントに委譲する

Agent ツールを `model: "sonnet"`、`run_in_background: false` で起動し、以下のテンプレートで指示する。`<操作リスト>` にはステップ1で具体化した操作を書く。

```
~/.claude/task-dashboard/ のタスクダッシュボードを更新してください。

## 操作
<操作リスト（例: 「開発環境」カテゴリに「- [ ] ○○」を追加 / 「××」の行を [x] にして 完了:日付 を付ける）>

## ルール
1. tasks.md を読み、上記の操作を適用する。
   - ステータス記号: [ ]=未着手 / [~]=進行中 / [!]=ブロック / [x]=完了。メタ情報は「| キー:値」形式
   - 完了は行を消さず [x] + 完了日。削除は明示指示があるときだけ
   - 日付は必ず `date +%Y-%m-%d` で取得（捏造しない）。frontmatter の updated も今日にする
2. dashboard.html を更新する。**再デザインせず、既存の <style> とHTML構造は保ったまま**、
   データ部分だけを tasks.md の全内容と一致するよう書き換える:
   サマリー数値（全件数・進行中・完了・達成率・進捗バーwidth）、カテゴリセクション、
   各タスク行（チップ: todo/doing/blocked/overdue、完了は done-box に）、最終更新日時（date "+%Y-%m-%d %H:%M"）。
   期限切れ（期限<今日）は overdue チップで警告表示。
   dashboard.html が存在しない場合のみ、artifact-design スキルを読み込んでゼロから生成する
   （<title>は「タスクボード」固定・自己完結・ライト/ダーク両対応）。
3. Artifact ツールで公開する。
   - **必ず tasks.md frontmatter の artifact_url を `url` パラメータに渡す**（渡さないと別URLが新規発行されてしまう）
   - artifact_url が空のときだけ新規公開し、返ってきたURLを frontmatter に書き戻す
   - favicon は「📋」固定、label は変更内容の短い語
4. 最終出力として、適用した操作・現在の集計（全件数/進行中/完了/達成率）・公開URLを返す。
   公開に失敗した場合は失敗と原因をそのまま返す（成功したことにしない）。
```

### 3. 結果を検証して報告する

サブエージェントの出力はユーザーに見えない。返ってきた内容から、変更の要約・現在のサマリー（例: 12件中5件完了・進行中2件）・ダッシュボードURLを自分の言葉で報告する。サブエージェントが失敗を報告したら、それを隠さず伝えてリカバリする（軽微ならメインが直接修正してよい）。

## 住み分け

- 中長期目標の分解・逆算 → `next-step`（分解した結果のタスク登録先としてこのスキルを使うのは可）
- GitHub Issue として起票したいもの → `issue-create`
- 実装セッション中だけの一時TODO → 本体の TaskCreate（このスキルは使わない）
