# Next Step — 目標を今日の一手まで分解する

抽象的な目標（「成長したい」「アウトプットを増やす」「Goを身につける」のような粒度の大きいもの）を、
**今日の最初の一手まで降ろす**ための逆算分解ガイド。

## このスキルの一番大事な原則

**分解ツリーを完成させることは成果物ではない。** 着手することが成果物。

このユーザーは内省・収集心が強く、最大の罠は「考えて終わる／出力しない」。
精緻な計画を作るほど、その計画づくり自体が新しい思索遊びになって着手ゼロで終わる。
だから毎回 **「今日やる最小の1個」＋「外部強制」** に必ず着地させる。ここを飛ばしたら失敗。

## ワークフロー

### Step 0. 目標を検証可能な1文に確定する

抽象的な願望を「**何を・どうする**」の1文にする。曖昧な動詞を観測可能な動詞に変換するのが肝。

| 曖昧（NG） | 観測可能（OK） |
|---|---|
| 理解する・身につける | 〜を見ずに書ける／人に説明できる |
| 頑張る・意識する | 週3回〜する／毎朝〜を出す |
| 成長する・うまくなる | 〜を3本公開する／レビューで指摘ゼロにする |
| 改善する | 〜の時間を30分→10分にする |

合意できた1文を冒頭に固定し、以降の分解はすべてこの1文に紐づける。

### Step 1. 完了条件（DoD）を先に決める ← 最重要

タスク分解より先に「**何が観測されたら達成と言えるか**」を事実で書く。
これを飛ばすと、分解が「もっと細かく…」と無限後退して計画倒れになる。

- 良い例：「Zenn記事が3本公開され、各記事に1件以上の反応がある」
- 悪い例：「Goが分かった状態」（観測できない＝終われない）

完了条件が複数あるなら、**1つに絞る**か、優先1つを主目標にする。

### Step 2. 「なぜ」に一度だけ戻る（手段の目的化チェック）

分解に入る前に1回だけ問う：**この目標は、本当に解きたい課題に対する手段になっているか？**

- 「AIを使いこなしたい」のような手段ファーストの目標は、理想→課題→手段の順に巻き戻す
- ただし「能力地図づくり（探索）」が目的なら、それはそのまま正当なゴールとして認める
- ここで目標自体が間違っていると気づいたら、Step 0 に戻ってやり直す

1往復で十分。ここを深掘りしすぎると、それ自体が思索遊びになる。

### Step 3. ゴールから逆算してマイルストーンに割る（2〜4個）

完了条件（ゴール）から**逆向きに**「その直前に何が起きているはず？」を辿り、中間地点を2〜4個置く。
前から積み上げない（前からだと「とりあえず勉強」のような着地しないタスクになる）。

- 各マイルストーンにも、観測可能な完了条件を1つずつ付ける
- 多すぎたら粗くまとめる。4個を超えるなら目標が大きすぎる → スコープを切る提案をする

```
ゴール: Zenn記事3本公開
  ← M3: 3本目を公開（残り完了条件…）
  ← M2: 2本目を公開
  ← M1: 1本目を公開
  ← M0: ネタ3つと構成を確定
```

### Step 4. 直近のマイルストーンだけタスク分解する

**全部のマイルストーンを分解しない。** いま着手する直近の1つ（上の例なら M0）だけを、
今日〜数日で着手できる粒度のタスクに割る。先のマイルストーンは見出しだけ置いて未分解で残す。

理由：先まで全部割ると「計画づくり」が肥大化して着手が遠のく。
直近だけ割れば、進んでから次を割れる（割り直しのほうが精度も高い）。

### Step 5. 最小の第一歩を1個 ＋ 外部強制を付ける ← ここで必ず着地

分解したタスクから、**今日の最初の15分でやる1個**を選ぶ。複数選ばせない。1個。
その1個に、次のうち最低1つの**外部強制**を必ず付ける（内省・「次から意識する」では弱い）：

- **締切**：いつまでに（日時で。「今週中」ではなく「金曜18時」）
- **他人の目**：誰に見せる／報告する（Slack・日報・DM）
- **役割**：「〜する人」として宣言する／場を予約する

外部強制が付けられない一手は、まだ大きすぎる。さらに小さく割る。

### Step 6. 優先順位（目標やタスクが複数あるとき）

複数の目標・タスクが並んだら **影響度 × 緊急度** で並べる。

- 影響度高×緊急度高：今日の一手にする
- 影響度高×緊急度低：締切を入れて外部強制で前に出す（放置で消える枠）
- 影響度低：やらない判断も提示する（全部やろうとしない）

### Step 7. 出力とハンドオフ

最後に、分解結果を構造化して提示する（ゴール1文 → 完了条件 → マイルストーン → 直近タスク → 今日の一手＋外部強制）。
そのうえで、行き先を提案する（強制はしない。判断はユーザー）：

- **複雑度2以上**（2ステップ以上／期限あり／調査が要る）→ `/issue-create` で Issue 化を提案
- **単発で終わる一手** → Issue にせず日報（`Vault/03_Journals/`）に書く運用を案内
- このスキル自体は Vault や Issue に書き込まない。提案までで止める

## トーンとスタイル

- 新卒に話すノリ、優しめ。事実と判断を返し、お世辞は言わない
- 1ターンに全ステップを詰め込まない。Step 0〜2（目標とDoDの確定）でいったん合意を取り、それから分解に進む
- 曖昧な目標が来たら、勝手に具体化せず「どっちの意味？」と1問だけ確認してから進める
- 絵文字は基本使わない。記号（← ✅ など）はOK
- 分解の途中で本人が「それより…」と別の本音を出したら、そっちが本当のゴールの可能性。Step 0 に戻る

## やってはいけないこと

- 完了条件（Step 1）を決める前にタスク分解を始める（無限後退の原因）
- マイルストーンを全部タスク分解する（計画倒れ・思索遊びの肥大化）
- 「今日の一手」と「外部強制」を出さずにセッションを終える（このスキルの最大の失敗）
- 外部強制を「次から意識する」のような内省で代用する
- 複数の一手を同時に持たせる（1個に絞る）
- 勝手に Issue を起票したり Vault に書き込む（提案までで止める）
- 「なぜ」の問い直し（Step 2）を1往復で切り上げず、延々と掘る

## 関連スキル

- `/issue-create`：分解した複雑度2以上のタスクを Issue 化する出口。このスキルの後段で呼ぶ
- `/structured-workflow`：実装タスクの設計→実行に特化。目標設定そのものではない
- `/progressive-learning`：「学ぶ」目標の中身を段階的に理解する用。目標分解とは別物
