# technical-note

会話で身についた技術的な学びを、いむたろ様の Obsidian Vault の **Technical ノート**として
「教えるのが上手い先輩エンジニア」のトーンで書き出すスキル。
コードや事実の丸写しではなく、**なぜ重要か → 何か → どう使うか**の流れで、後から読んだ本人が
再現できるレベルにまとめる。

> ⚠️ このスキルは Vault（`Vault/01_Notes/`）に**直接書き込む**。本来この Vault は「AI は読むだけ・
> 書き込み禁止」が原則（中核思想＝AI は自分の言葉で Vault を希釈しない）。いむたろ様が「Technical 直書き OK」と
> 明示合意した上での**異例運用**なので、後述の透明性マーカーと「自分の言葉で見直す」リマインドを必ず守る。

## 起動条件

いむたろ様が明示的に依頼したときだけ起動する（自動起動しない）。代表的なトリガー：

- 「（いまの話を）テクニカルにまとめて」「Technical ノートにして」
- 「Obsidian のテクニカルに保存して」「この概念をテクニカルノート化して」
- 「○○を Technical に」

## ネタ元（インプット）の制約

**次の2つだけをネタ元にする。これ以外から内容を足さない。**

1. **直近の会話履歴** — いまのセッションで Claude が説明した／話題に出た技術概念
2. **明示的に渡されたコード/ファイル** — パスやコードスニペットを渡されたら、それを読んで解説する

やらないこと：

- 勝手な Web 調査・外部リサーチ（必要だと思っても、足りない箇所は本文に「未確認」と書いて止める）
- `Vault/01_Notes/` の既存ノートの**書き換え**（このスキルが触るのは新規 or 明示指定の Technical ノートだけ）
- ネタ元が曖昧なときの推測。何をまとめるのか1問だけ確認してから書く

## 出力仕様

### 置き場所（固定）

```
/Users/imutaakihiro/repos/imutaakihiro/ObsidianImus/Vault/01_Notes/Technical/Technical - {トピック}.md
```

> このパスはハードコード。別マシン／パス変更で壊れる前提（パーソナル運用なので許容）。

### ファイル名

- 必ず `Technical - ` プレフィックスで始める
- 大分類があるトピックは `Technical - {大分類} - {トピック}.md` 形式にする
  （例：`Technical - BigQuery - ウィンドウ関数.md`、`Technical - Github - コンフリクトの原因と対策.md`）
- 書く前に `Technical/` の既存ファイル一覧を確認し、命名の流儀（既存の大分類）を踏襲する
- **同名ファイルがあったら上書きしない。**「追記するか・別名にするか」を1問確認する

### frontmatter（最小・JIT）

```yaml
---
type: technical
tags:
  - technical
created: {YYYY-MM-DD}   # date コマンドで取得した当日
source: claude-draft    # AI が会話から下書きした透明性マーカー（自分の言葉で見直したら外してよい）
---
```

### 本文の型（Why / What / How）

Technical テンプレートに準拠。内容に応じて見出しを足してよいが、この3つを軸にする。

```markdown
### Why
なぜこれを知る必要があるか・どんな場面で効くか（背景・動機）

### What
それは何か（定義・仕組み・要点）。必要ならコードブロックや表

### How
どう使うか（手順・具体例・つまずきどころ）
```

### トーン・作法（01_Notes の原則を厳守）

- **自分の言葉で**。会話やコードの丸写しをしない。要点を噛み砕いて書く
- **リンクで繋ぐ**：関連する既存ノートがあれば `[[Vault/01_Notes/Technical/Technical - xxx|表示テキスト]]` でリンク。
  実在を確認できたノートだけリンクする（存在しないノートへ無闇にリンクしない）
- 簡潔に。冗長な前置きや一般論で埋めない（既存 Technical ノートは短くて濃い）
- Callout を使うなら `[!info]` `[!warning]` `[!success]` の3種のみ

## 手順

1. **ネタ元を確定**：会話履歴か、渡されたコード/ファイルか。曖昧なら「何を Technical にまとめますか？」と1問だけ聞く
2. **当日の日付を取得**：`date +%Y-%m-%d`
3. **既存一覧を確認**：`Vault/01_Notes/Technical/` を `ls`/Glob で見て、命名の流儀と重複をチェック
4. **同名があれば確認**：追記 or 別名をいむたろ様に確認
5. **関連ノートを軽く検索**：リンクできる既存ノートがあるか Grep/Glob で探す（あれば本文にリンク）
6. **本文を組み立て**：Why / What / How。frontmatter を付ける
7. **直接 Write**：上記パスに `Technical - {トピック}.md` を書き出す
   - 長くなる場合は骨子を `Write` → 各節を `Edit` で継ぎ足す（一発 Write でツール呼び出しが壊れるのを防ぐ）
8. **完了報告**：書き出したパス・見出し構成・貼った関連リンクを伝える。
   最後に必ず一言：「**自分の言葉で一度見直すと第二の脳に定着します**（`source: claude-draft` は見直したら外してOK）」

## 完了報告のフォーマット例

```
✅ Technical ノートを作成しました
- パス: Vault/01_Notes/Technical/Technical - ○○.md
- 構成: Why / What / How
- 関連リンク: [[...]]（n本）

自分の言葉で一度見直すと第二の脳に定着します。
（AI 下書きマーカー source: claude-draft は、見直したら外してOK）
```
