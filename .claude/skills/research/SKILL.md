# リサーチ（research）

## 前提

このSkillは ObsidianImus Vault プロジェクトの構造を前提にする：

- **出力先**: `Company/Drafts/YYYY-MM-DD-{slug}.md`（1案件=1ノート）
- **書き込み禁止**: `Vault/` 配下（読むだけ）

---

## ワークフロー（6ステップ）

### Step 1: リサーチ目的の確認

`AskUserQuestion` で1問だけ確認：

> このリサーチの目的はどれが近いですか？

| 目的 | 軸の方向性 |
|------|-----------|
| **A. 事前準備**（イベント・場所・人物・体験） | 多方面から「行く・会う・体験する」前の準備情報 |
| **B. 学習・知識化**（テーマ理解） | 概念・歴史・実例・反論・応用の5軸 |
| **C. 競合・市場分析** | 競合・市場規模・トレンド・ベストプラクティスの軸 |
| **D. 意思決定の材料集め** | メリット・デメリット・代替案・実例の軸 |

→ ご主人様が選んだ目的に応じて Step 2 の軸を組む。

### Step 2: リサーチ軸の提示

目的に応じて5軸を提示し、承認を得る。

#### A. 事前準備テンプレ（行天リサーチが該当）
1. 基本情報（場所・時間・予算・予約難易度）
2. 中心人物（経歴・哲学・印象的な言葉）
3. 業界での位置づけ（評価・実績）
4. 作法・マナー
5. 体験者の声・印象的なエピソード

#### B. 学習・知識化テンプレ
1. 定義・概念
2. 歴史・背景
3. 実例・ケーススタディ
4. 反論・批判的視点
5. 自分の文脈への応用

#### C. 競合・市場分析テンプレ
1. 競合リスト
2. 市場規模・トレンド
3. 各社の差別化ポイント
4. 顧客の評価・不満
5. 自分の打ち手

#### D. 意思決定テンプレ
1. メリット
2. デメリット・リスク
3. 代替案
4. 実例（成功・失敗）
5. 判断基準

→ 軸を変更したい場合はご主人様の指示で組み直す。

### Step 3: 並列調査（最重要）

#### 3-1. ツール準備
`WebSearch` と `WebFetch` のスキーマがロードされていなければ、`ToolSearch` で先に取得：
```
ToolSearch(query: "select:WebSearch,WebFetch", max_results: 2)
```

#### 3-2. 並列実行
**5本の WebSearch を1メッセージで並列発射**する（重要：直列にしない）。

検索クエリは「強い言葉」を意識：
- OK: 「徹底的に」「深く」「詳細に」「完全ガイド」
- 禁止: 「ちょっと」「概要」「ざっと」

#### 3-3. リアルタイム報告
調査プロセスを簡潔に報告：
- 何を見つけたか
- 何が浅いか
- 追加調査が必要な箇所

### Step 4: 補完調査

並列調査で浅かった箇所を1〜3本追加で深掘り：
- 中心人物のインタビュー記事を WebFetch
- 体験者の生の声を WebFetch
- 公式情報の確認

### Step 5: レポート生成

#### 5-1. 規模判定
| 規模 | 目安 | 出力先 |
|------|------|--------|
| **単発** | 成果物1ファイル / 1〜2時間 / 問い1つ | `Company/Drafts/YYYY-MM-DD-テーマ.md` |
| **案件** | 成果物複数 / 複数セッション / 派生あり | `Company/Drafts/YYYY-MM-DD-テーマ/` |

#### 5-2. 案件フォルダ構造（案件の場合）
```
Company/Drafts/YYYY-MM-DD-テーマ/
├── 00_brief.md    目的・完了条件・参考URL
├── 10_input/      1次資料・調査対象
├── 20_wip/        中間メモ
└── 30_output/     整理済み成果物
```

#### 5-3. レポート構成・frontmatter（必須）

frontmatterは現行標準（`type: ai-work` / `status` / `created` / `topic` / `promote`）に従う。リサーチ特有の情報（目的タイプ・依頼背景など）はfrontmatterに含めず、本文冒頭の `## リサーチ概要` セクションに書く：
```markdown
---
type: ai-work
status: review
created: YYYY-MM-DD
topic: 一言で何の調査か
promote: なし
---

## リサーチ概要
- 目的タイプ: 事前準備 / 学習・知識化 / 競合・市場分析 / 意思決定
- 依頼背景: ...

> [!info]
> 秘書からのメッセージ（このリサーチで一番伝えたいこと）

## 0. 30秒サマリー
| 項目 | 内容 |
|------|------|
| ... | ... |

## 1. [軸1の見出し]
[詳細]

## 2. [軸2の見出し]
[詳細]

...

## N. 出典
- [タイトル](URL)
- ...
```

成果物が揃ったら `status: review` にする。`AI Desk.base` の「🟡 レビュー待ち」タブに自動表示される（Issueコメント等は不要）。

### Step 6: 完了報告

ご主人様に以下を報告：

1. **ファイルパス**（クリックで開ける形）
2. **30秒サマリー**（最重要発見3つ）
3. **次のアクション提案**（あれば）：
   - 実体験への接続
   - note記事化（`/note-create` 連携）
   - 追加調査の必要性

---

## 重要な注意事項

### 禁止: やってはいけないこと
- 直列での WebSearch（必ず並列5本）
- 出典の省略（必ず markdown リンクで列挙）
- `Vault/` 配下への書き込み
- 表面的な「ざっと調査」（強い言葉でクエリを組む）

### OK: 守るべきこと
- 並列実行を最優先（時間効率）
- ご主人様の文脈に接続するセクションを必ず1つ入れる
  - 例：「社会人一年目のご主人様にとっての意味」「グロースハック部署への接続」
- 出典は必ず明記（信頼性の担保）
- 30秒サマリーを冒頭に必ず置く（焦って読むときのため）

### ヒント: 推奨パターン
- 「リサーチ → note記事化」のワークフローを意識（`/note-create` への接続）
- 案件リサーチは `00_brief.md` で目的・完了条件を最初に明記
- 完了時は秘書メッセージで「このリサーチで一番伝えたいこと」を1段落
