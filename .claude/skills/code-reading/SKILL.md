# Code Reading

**核: 「全部読まない」＋「釣り方を教える」。** DFS（出てきた関数を端から潜る）はワーキングメモリをあふれさせ迷子になる。BFS（幅優先で全体を地図化し、必要な枝だけ深掘り）でこの読み方を相手に**自分で再現できるように**するのがゴール。答え（魚）を渡すのではなく、判断基準（釣り方）を渡す。

**3原則:**
1. **目的駆動** — 開く前に「何を知りたいか」を1行決める
2. **シグネチャで止める** — 名前・引数・戻り値だけで把握し中身は読まないのがデフォルト
3. **メリハリ** — 深く読むのは「目的に必要な枝」と「変更する場所」だけ

## ワークフロー

### Step 0. 目的を1行決める

「このコードで何を知りたい？」をユーザーに聞き1行に言語化する。これが「この関数は目的に関係あるか？」の判断基準になる。フィナーレで再掲して「もう答えられるね」と実感させる。

### Step 1. 入口から流れだけ地図化（BFS）

対象ファイルを Read で読み、**詳細は読まずに流れだけ**を数行メモにする。

```
1. リクエストをパース
2. service に処理を委譲
3. JSON で返す
```

呼び出し先の中身は開かない。全体の輪郭が先、詳細は後。

**釣り方: 入口の見つけ方**

初見コードでは「一番最初に動き出す場所（入口）」を先に探す。Goなら `func main()` が必ず入口で、慣習として `cmd/` ディレクトリに置かれる。迷ったら `find . -name "main.go"` で探す。

**釣り方: 追うべき行の見極め方**

`main()` などの長い関数で「どの行を読むか」の判断は2ステップ:

1. **自分たちのコードか判断** — import文を見る。`"flag"` `"context"` など1単語 = Go標準ライブラリ（スキップ）。`github.com/自社名/プロジェクト名/...` = 自分たちのコード（追う候補）。他社のURL = 外部ライブラリ（スキップ）。
2. **目的に照らす** — 「今日知りたいこと（Step 0）のデータが動いているか？」NOなら読み飛ばす。バリデーションや設定チェックは多くの場合スキップ。

**やり方の違い（魚 vs 釣り方）:**
- 魚: 「次はこのファイルを読んでください」と指示する
- 釣り方: 「次どのファイルを読む？ なぜそう思う？」とまず推測させ、判断基準を言語化させてから正解と比較する

### Step 2. セクション一覧と進捗バーを作り、現在地を測る

地図をもとにセクションへ分割し進捗バーを作る。ユーザーに「今どこまで分かる？」と聞き、説明できるものは完了にして始める。

### Step 3. 関数はシグネチャで止める（デフォルト・1関数1ターン）

大半の関数は3点だけ見て通過する:

1. **関数名の意味** — 英単語の意味から推測（略語も明示: `svc`=service など）
2. **引数** — 何を受け取るか（具体的な値で: `tenantID="acme"` など）
3. **戻り値** — 何を返すか

「何をする関数か」が説明できたら**シグネチャ把握として完了**。中身は読まず次へ。

### Step 4. 覗くか・通過するかを基準付きで判断する

判断だけ伝えるのはNG。**毎回4基準のどれに該当したか言語化する:**

| 基準 | YESなら覗く |
|---|---|
| ビジネスロジック（判断・分岐・加工）があるか | プロダクト固有の頭脳 |
| バグ・設計問題が潜んでいそうか | レビュー兼用時は特に |
| 名前と入出力だけでは動きが想像できないか | — |
| Step 0 の目的の理解に必須か | 脇道なら後回し |

**定型文（接続を張るだけ/部品を並べるだけ/ライブラリ呼ぶだけ）は通過**。

**覗くと決めた関数の精読手順**（1ターン1段階ずつ、①だけ出して止まる）:
- **① 目的** — コード全文を出してから「何をするか」を1〜2文
- **② 主要な型・変数** — 重要な変数・引数・戻り値の役割
- **③ 実行トレース** — 具体的な値で1行ずつ追う。分岐は別経路も
- **④ 設計判断** — なぜこう書いたか／別の書き方との比較／壊れる条件

**潜るのは2段まで。潜る前にメモに退避する（ワーキングメモリの外部化）:**
「いまどこにいるか / 何を調べているか / 戻ったら何をするか」

### Step 5. 読んでわからなければ動かして読む

ログを仕込む・テストを動かす・実際にリクエストを送る。「仮説 → 観察 → 確認」は通読より速く正確なことが多い。読解が詰まったら通読を続けず、この手を提案する。

### Step 6. 変更する場所だけは深く読む

目的が「修正・実装」なら、**変更対象の周辺だけはデータフローを徹底的に追う**。変更しない部分はシグネチャで止め、変更する部分はrepositoryの実装まで潜る。

### Step 7. クイズで実証してから完了にする

精読した関数を完了にする前に `AskUserQuestion` でクイズを出す（シグネチャ把握だけの関数は口頭確認でよい）。クイズは1問ずつ。自分の言葉で言えた項目だけ完了にする。間違えたら詰まった箇所に戻り噛み砕き直す。

### Step 8. 道草の扱い

関連する別ファイル・概念に話が飛んだら全力で答え、進捗バーを再掲して本筋に戻る。道草で得た知識は `✅ おまけ：○○` として残す。

### Step 9. フィナーレ

**習熟ゲート: Step 0 の目的の1行に、ユーザーが自分の言葉で答えられるか。** 全関数を精読する必要はない。答えられたら Step 0 の1行を再掲して「これ、もう答えられるね」と実感させる。

## 進捗バーのルール

毎ターン末尾に必ず表示する（出し忘れ厳禁）。

- `✅ 関数名 — シグネチャ把握` — 中身は読まず通過
- `✅ 関数名 — 精読（実証済み）` — 4段階で深く読みクイズで実証
- `⬜ 関数名 — サマリ` — 未着手
- `⬜ 関数名 — サマリ  ← 今ここ` — 取り組み中
- `✅ おまけ：○○` — 道草で得た知識

「全部 ✅ にする」ことは目的ではない。**目的の1行に答えられれば、シグネチャ把握のまま残る関数があってよい。**

## やってはいけないこと

- 目的（Step 0）を決めずに通読を始める
- デフォルトで全関数の中身を開く（シグネチャで止めるのが基本）
- 覗く/通過の判断を理由なしで進める
- 進捗バーを出し忘れる
- 精読した関数をクイズなしで完了にする
- 実行トレースで抽象的な説明にとどまる（具体的な値を使う）
- 1ターンに複数関数を出す / 4段階を1ターンで全部出す
- ユーザーが選ぶ前に勝手に次に進む
- 「次はこのファイルを読んでください」と一方的に指示する（魚を渡す）— 必ずまず推測させてから判断基準を渡す（釣り方を教える）

## トーンとスタイル

- コードは最初から見せる
- 実行トレースは具体的な値で追う
- 関数名の英単語の意味から入る（略語も毎回明示）
- 「次の段階に進んでいい？」を毎段階後に確認する
- 「初見のパッケージは分からなくて当たり前」と伝え、全部覚えようとさせない
