# Feedback → Slack Formatter

メンターとの 1on1 / 日報フィードバックの**話し言葉の文字起こし**を、**相手（メンター）の発言を中心に**整理し、Slack にそのまま貼れるフォーマットへ整形する。
姉妹スキル `daily-report-formatter`（自分の発言→日報）とは逆向き。**こちらは相手の発言を拾う**のが核。

## 最重要原則：話者の分離

- 文字起こしには「ご主人様（本人）」と「相手（メンター）」の発言が混在している。
- **抽出するのは相手＝メンターの発言（フィードバック・助言・指摘・論点整理）だけ**。
- 本人の発言は、相手の発言を理解するための**文脈**としてのみ使い、まとめ本文には載せない。
- 文字起こしが崩れていて話者が判別しづらい場合は、内容（フィードバックを「与える」側か「受ける」側か）から推定する。判別不能な箇所は最後の確認ステップで聞く。

## ワークフロー

1. **対象日付を確定**
   - ユーザーの言及から推定（例「22日の日報のFB」→ 今月の22日）。
   - 言及がなければ `date +%Y-%m-%d` で今日を仮置きし、確認する。
2. **話者を分離**し、相手（メンター）の発言を抽出する（上記「最重要原則」に従う）。
3. **テーマごとに動的にセクション化**する（固定見出しは使わない。下記「セクション構成」参照）。
4. **Slack 記法で整形**する（下記「出力フォーマット」「ライティングルール」）。
5. **チャットにコードブロックで出力**する（Slack 記法を素のまま見せ、コピペできるように）。
6. **日報ファイルの末尾に追記保存**する（下記「保存先」）。
7. **確認ステップ**：聞き取りが怪しい固有名詞と、投稿の宛先／トーンをユーザーに確認する。

## セクション構成（動的）

文字起こしの中身からテーマを抽出して見出しを立てる。テーマ数・名前は内容しだいで変える。
代表的なテーマ例（毎回これに固定しない）：

- 進め方・コミュニケーションへのフィードバック
- 具体的な指摘・改善点
- プロジェクト／業務の論点整理（相手と一緒に整理した内容）
- キャリア・スタンスへの助言
- 次のアクション・宿題

各セクションは「絵文字 + 見出しテキスト」→「`•` の箇条書き」で構成する。

## 出力フォーマット

Slack 記法。チャットには ` ``` ` のコードブロックに入れて出力し、そのままコピーできるようにする。

```
:memo: {日付} {テーマ}フィードバックまとめ

:dart: {テーマ見出し1}
• {相手の発言の要点}
• {特に重要な指摘もそのまま箇条書きで}

:warning: {テーマ見出し2（指摘・注意系）}
• {内容}

:bulb: {テーマ見出し3（論点整理系）}
1. {番号付きが合う場合は番号で}
2. {内容}
• {補足}

:seedling: {テーマ見出し4（助言系）}
• {内容}
```

絵文字は内容に合うものを選ぶ（例：`:dart:` 進め方 / `:warning:` 指摘 / `:bulb:` 論点 / `:seedling:` 助言 / `:speech_balloon:` 所感 / `:white_check_mark:` 次アクション）。見出しと箇条書きの構造だけで読ませる（太字に頼らない）。

## ライティングルール

- **Slack 記法を厳守**する：
  - **太字（`*…*` / `**…**`）は使わない**。ご主人様の Slack 環境では反応しないため。強調は絵文字見出しと箇条書きの構造で表現する。
  - 箇条書きは `•`（`-` や `*` を行頭に使わない）。
  - 見出しに `#` / `##` は使わない（Slack では効かない）。絵文字 + 見出しテキストで代用。
  - 絵文字は `:dart:` のようなコロン記法で書く。
- **相手の発言の要点のみ**を書く。本人の発言・地の文は混ぜない。
- **文字起こしにある内容だけ**を使い、捏造しない。聞き取れない・確信が持てない箇所は省き、確認ステップに回す。
- 似たトピックは**意味で束ねて**1テーマに統合し、見出しを増やしすぎない（目安 3〜5 テーマ）。
- 各箇条書きは要点を1〜2文で簡潔に。スキャンできる長さにする。
- **FBer（相手）の名前は原則タイトル・本文に入れない**（汎用化のため）。ご主人様が「名前を入れて」と明示したときだけ `（{名前}より）` を付ける。
- プロダクト名・ツール名などの固有名詞は文字起こしの音から最も妥当な表記を当て、不確実なものは確認ステップで挙げる。

## 保存先

- Vault: `/Users/imutaakihiro/repos/imutaakihiro/ObsidianImus/`
- ファイル: `Vault/03_Journals/{対象日付}.md`（日報本体と同じファイル）
- **末尾に追記**する（既存の日報内容は消さない）。Slack 記法は Obsidian で崩れるため、追記ブロックは**コードフェンスで囲って**貼り付け忠実性を保つ。

追記する形：

```
（↑ 既存の日報内容はそのまま）

## :speech_balloon: メンターFB（Slack用 / {対象日付}）

` ` `
{Slack記法の整形済みテキスト}
` ` `
```

- 対象日付の日報ファイルが存在しない場合は、勝手に新規作成せず、ファイルの有無と保存先をユーザーに確認する。
- これは `morning` / `daily-report-formatter` と同様、`Vault/03_Journals/` への**AI 書き込みが許可された例外**スキル。日報の本文（自分の言葉部分）は書き換えず、末尾に FB ブロックを足すだけにする。

## 確認ステップ（保存後）

整形・保存が終わったら、最後に次を確認する：

1. **聞き取りが怪しい固有名詞**（プロダクト名・ツール名など。人名は原則本文に入れないので確認不要）を列挙し、表記の正否を聞く。
2. **投稿の宛先**（自分用ログ / 相手へのお礼＋要約 / チーム共有）を聞き、必要ならトーン（敬語の硬さ）を調整する。
3. 文脈が曖昧で本文から外した発言があれば、入れるか聞く。
