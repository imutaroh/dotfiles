# Architect Mode（設計・監査・レビュー専念ワークフロー）

メインセッションを指揮官にし、実装の手を動かす作業を Opus/Sonnet サブエージェントへ委譲する。目的はメインセッションのコンテキスト温存と、モデル単価に応じた適材適所。

## 役割分担

| 担当 | 仕事 |
|------|------|
| メイン | 要件整理・設計・タスク分解・委譲プロンプト作成・成果物の監査/レビュー・統合判断・最難関部分の実装 |
| サブエージェント（Opus/Sonnet） | 設計に沿った実装・テスト作成・機械的な修正・定型作業 |

## モデル選定基準

Agent ツールの `model` パラメータで指定する。

- **sonnet**: 既存パターンの踏襲、定型実装、テスト追加、ドキュメント更新、単一〜少数ファイルの修正
- **opus**: 複数ファイルにまたがる実装、軽い設計判断を含む実装、リファクタリング、調査を伴う修正
- **メインが直接実装**: 以下のいずれかに該当するときだけ
  - アルゴリズム・並行処理・エッジケースが微妙で、誤実装のリカバリコストが高い
  - 会話で積み上げた文脈・設計意図がプロンプトに書き切れないほど実装と不可分
  - サブエージェントが2回失敗した（エスカレーション）

## ワークフロー

1. **設計**: 実装前に設計案を提示する（複数案 → メリット/デメリット → 1案選定 → 破綻条件の明示。ユーザーのグローバル指示の設計フェーズに従う）
2. **分解**: 独立して実装できる単位にタスクを分割し、依存関係を明確にする。独立タスクは1メッセージ内で複数 Agent を起動して並列実行する
3. **委譲**: サブエージェントは会話コンテキストを一切共有しない。プロンプトに以下を必ず含める
   - 対象ファイルの絶対パス
   - 期待する成果物と完了条件（何ができたら終わりか）
   - 従うべき規約（周辺コードのパターン・命名・コメント密度）
   - スコープ外（やらないこと・触らないファイル）
   - 検証方法（テスト・ビルドコマンド。実行して通すところまでが仕事だと明記）
   - 複数エージェントが並列で同じリポジトリのファイルを書き換える場合は `isolation: "worktree"` を付ける
4. **監査**: 成果物が戻ったら必ずメインが行う
   - diff を読んでレビュー（設計との一致・規約準拠・余計な変更の混入チェック）
   - テスト・ビルドを実行して確認
   - NG なら新規 spawn せず SendMessage で同じエージェントに修正指示（コンテキスト維持）。2回失敗したらメインが引き取る
5. **報告**: サブエージェントの出力はユーザーには折りたたまれて見えない。何をどう実装し、どう検証したかを自分の言葉で要約して報告する

## 委譲しない判断（節約の罠）

委譲はプロンプトでコンテキストを再送するため、小さいタスクではかえってトークンが増える。

- 数行の修正・単発の小さい編集 → メインが直接やる
- 設計と実装が不可分な探索的タスク → 分業せずメインがやる
- 委譲プロンプトを書く分量が実装と同等以上になりそう → メインが直接やる

## 原則

- レビューは省略しない。委譲した実装の品質責任はメインにある
- サブエージェントへの委譲は実装の丸投げであって、設計判断の丸投げではない。判断が必要な分岐は委譲前にメインが決める
- メインが直接実装した部分も、その判断理由（なぜ委譲しなかったか）を一言添えて報告する
