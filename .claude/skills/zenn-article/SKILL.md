# Zenn記事作成（zenn-article）

## 前提

- **出力先リポジトリ**: `/Users/imutaakihiro/repos/imutaakihiro/ObsidianImus/zenn-claude-guide/articles/`（Zenn連携済み・private）
- **公開の主導権はユーザー**: このスキルは常に `published: false` で書く。`true` への変更はユーザーが明示的に指示したときだけ。push しても Zenn 上では「下書き」扱いになるのはこのため
- **git 操作は記事ファイル単独**: このリポジトリには他の作業中ファイルがあることが多い。`git add` は作成した記事ファイルだけを指定し、`git commit -- <ファイル>` で巻き込みを防ぐ

## Step 1: ネタとタイプを確定する

入力は3パターンある。どれかを判定して素材を集める:

1. **会話履歴**: 直近のセッションでやった開発・構築・調査（このパターンが最多。実際に起きたことだけを書く）
2. **既存ドラフト**: Obsidian やファイルのドラフトを指定された場合は Read する
3. **口頭のネタ**: 材料が薄い場合、記事の柱になる「実体験・ハマりポイント・数字」を1〜2問だけ質問して補う

タイプを決める: 手を動かした記録・手順・検証 → `tech`。意見・考察・ポエム寄り → `idea`。迷ったら tech。

## Step 2: 骨子を提案してから書く

いきなり全文を書かず、タイトル案（2〜3個）+ 見出し構成を先に見せて方針を合わせる。
ただしユーザーが「そのまま書いて」「一気に」系の指示をした場合や、素材が会話履歴で十分に濃い場合は、骨子提案をスキップして執筆に進んでよい。

## Step 3: 執筆する

### Zennで読まれる記事の型

- **冒頭3行がすべて**: 抽象的な前置き（「〜について書きます」）で始めない。具体的な瞬間・結果・数字から入る。
  - 例: 「『よし、ブログ作ろう』と思い立った日の夜、本番URLをスマホで開いていました。」
- **数字で具体化**: 「簡単だった」ではなく「コード0行」「実働半日」「10分溶かした」。タイトルにも最強の数字を1つ入れる
- **ハマりポイントを正直に書く**: 成功談だけの記事は信頼されない。エラーメッセージの実物・原因・回避策をセットで
- **短段落**: 1段落は2〜3文まで。見出しを細かく割ってスクロールのリズムを作る
- **まとめは箇条書き4点以内 + 最後に一言の主張**で締める

### Zenn固有の規約（frontmatter とファイル名）

```yaml
---
title: "タイトル（70字以内目安。数字か具体ワードを入れる）"
emoji: "📝"        # 1文字だけ
type: "tech"       # tech / idea
topics: ["microcms", "nextjs"]  # 最大5つ。小文字英数字。Zennで使われている表記に合わせる
published: false   # このスキルでは常に false
---
```

- **ファイル名（slug）**: `a-z0-9` とハイフン・アンダースコアで **12〜50文字**。短すぎると Zenn 側でエラーになる（例: `microcms-nextjs-easy-blog.md`）
- **Zenn記法を活用**: 補足は `:::message`、長い脱線は `:::details タイトル`、コードブロックは言語指定必須
- **画像**: リポジトリの `/images/` に置いて相対参照（`/images/xxx.png`）。外部URL直貼りは避ける

### 書いてはいけないもの

- 実体験にない手順・数字の捏造（「たぶんこうなる」を体験のように書かない）
- APIキー・サービスID・トークン類。実物のIDはすべて `xxxx` にマスクする
- 誇張タイトル。数字は実測ベースにし、AI並走など前提があるなら本文で開示する

## Step 4: 配置してZennの下書きに反映する

1. `articles/<slug>.md` に保存
2. リポジトリの `git status` を確認し、**作成した記事ファイルだけ**をコミット:
   ```bash
   git add articles/<slug>.md
   git commit -m "<記事タイトル>の下書きを追加" -- articles/<slug>.md
   git push origin HEAD
   ```
3. push 後、「Zennのダッシュボード( https://zenn.dev/dashboard )に下書きとして反映される」ことをユーザーに伝える
4. Obsidian（`ObsidianImus/Company/Drafts/`）へのミラー保存は、ユーザーに求められた場合のみ

## Step 5: 完了報告

以下を簡潔に報告する:

- 記事ファイルのパスと slug
- タイトルと構成（見出しリスト）
- 公開手順の案内: 「公開するときは frontmatter を `published: true` にして push、または指示をください」
- 事実確認を促す: 数字・手順・固有名詞で自信がない箇所があれば、正直に列挙してユーザーの確認を仰ぐ
