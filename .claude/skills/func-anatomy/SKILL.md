# Func Anatomy — 関数の解剖図を一発で出す

対象の関数を実コードに基づいて解剖し、固定フォーマット一画面で解説する。
目的は「これを見たら論理構造が理解できる」状態を最短で作ること。教育的な出し惜しみはしない。

## ワークフロー

### 1. 対象の特定

- 関数名・シグネチャ・コード片のいずれかを受け取り、Grep で定義位置を特定する
- 同名関数が複数ヒットしたら、ファイルパスを列挙してどれかを確認する（推測で選ばない）
- 定義が見つからなければ正直にそう言う。想像で解剖しない

### 2. 材料の収集（読む範囲のルール）

- **対象関数の本体**: 全部読む
- **呼び出している関数**: シグネチャ＋doc comment まで。本体は原則読まない。
  ただし数行で終わる小さな関数や、名前から役割が推測できないものは本体まで読む
- **呼び出し元**: Grep で検索し「誰が・どういう文脈で呼ぶか」を掴む
- **引数・戻り値の型**: 構造体ならフィールド定義を確認する
- 展開は **1 階層まで**。呼び出し先の呼び出し先までは潜らない（潜りたければ深掘り選択肢に回す）

### 3. 解剖図の出力（固定フォーマット）

以下の順・見出しで出す。すべて実コード根拠（`file.go:行番号` を添える）。想像や一般論で埋めない。

```markdown
## 一言でいうと
（この関数が何をするか 1〜2 文。「誰が呼ぶか」も含める）

## シグネチャ分解
| 部位 | 型 | 意味 |
|---|---|---|
| レシーバ/引数/戻り値を1行ずつ | 型 | 何を表すか・どこから来るか/どこへ行くか |

## 処理の流れ
1. 番号付きステップ。分岐・早期 return・ループを明示（行番号つき）
2. ...

## 呼び出している関数
| 関数 | どこの層/パッケージ | 役割 | 入力 → 出力 |
|---|---|---|---|

## 失敗するとどうなる
（エラーパス: どこで失敗しうるか、失敗時に呼び出し元へ何が返るか、ログ/HTTPステータス等の副作用）

## 呼び出し元
（誰が・いつ呼ぶか。エントリポイントからの経路を1行で: 例 Scheduler → Handler → この関数）
```

### 4. 深掘りの選択肢（最後に必ず）

解剖図の後に、次に掘れる場所を 2〜3 個提示して相手に選ばせる（例:「呼び出し先 X の中身」「この型の定義」「エラー時のリトライ経路」）。
本編は全部出す・深掘りは選ばせる、の二段構え。

## ルール

- **一画面主義**: 本編はスクロール 2 画面以内。収まらないなら流れのステップを束ねて粒度を上げる（情報を削るのではなく抽象度で調整）
- **表の中は結論だけ**、説明が要るものは表の外の文で補う
- 比喩・例え話は使わない（それは learn / progressive-learning の担当）。事実と構造だけで理解させる
- 対象がメソッドチェーンやコールバックを含む場合、「いつ実行されるか」（即時か、後で呼ばれるか）を処理の流れの中で明示する
- 言語固有のイディオム（Go の `defer`、早期 return 等）が流れの理解に必要なら、1 行だけ補足を入れる
