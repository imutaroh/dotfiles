# 日本語テクニカルライティング基礎規範

読みにくい技術文書の原因の大半は、内容の難しさではなく、文の構造と表記の乱れである。
この規範は、意味が一読で確定する文を書くための最低条件を定める。文章に緩急を付ける規範（`../cognitive-rhythm-writing/SKILL.md`）は、この土台が満たされた文章にだけ適用する。

## 文の構造

- **一文一義**。一つの文で伝えることは一つにする。「〜で、〜が、〜ので、〜です」と接続助詞で繋いだ文は、意味の切れ目で分割する。目安として、一文は80字を超えない。
- **主語と述語を近づける**。主語と述語の間に長い修飾節を挟まない。挟みたい情報は文を分けて先に置く。
- **係り受けを一意にする**。修飾語がどの語に係るか二通りに読める文は書き直す（「速いGoのコンパイラ」→「Goの速いコンパイラ」）。読点で解決しようとせず、語順で解決する。
- **二重否定を使わない**。「〜できなくはない」「〜しないわけではない」は肯定形に直す。
- **受動態より能動態**。動作の主体が明らかな文は能動態で書く。主体が本当に不明・不要な場合だけ受動態を許す。
- **並列は形を揃える**。「AすることとB」のような体言と用言の混在をなくし、箇条書きの各項目は文末の形（体言止め/です・ます/である）を統一する。

## 冗長表現の削除

以下は機械的に置換する。

- 「〜することができる」→「〜できる」
- 「〜という機能」「〜というもの」→「という」を削る（引用・命名の導入を除く）
- 「〜を行う」「〜を実施する」→ 動詞そのもの（「検証を行う」→「検証する」）
- 「〜のような形で」「〜といった形に」→「〜のように」または削除
- 「基本的に」「一般的に」「ある意味」→ 例外を本当に説明する場合以外は削除
- 「まず最初に」「あらかじめ予約」などの重複表現 → 片方を削る

削った結果、文が事実の羅列だけになっても構わない。飾りを足して埋めない。

## 用語と表記

- **用語は文書内で一意にする**。同じ対象を「サーバ/サーバー」「ディレクトリ/フォルダ」「引数/パラメータ」と揺らさない。初出時に採用形を決め、以降は固定する。
- **英数字は半角**。全角英数字・全角記号（（）：！？を除く和文用記号以外）は使わない。和文と英単語の間にスペースは入れない方針か入れる方針かを文書内で統一する。
- **カタカナ語の長音**は製品・言語の公式表記に従う（例: 「サーバー」「ユーザー」）。公式表記がないものは長音ありで統一する。
- **漢字を開く**：形式名詞・補助動詞はひらがなにする。「事→こと」「時→とき」「〜して下さい→してください」「〜する為→ため」「出来る→できる」「更に→さらに」。固有の意味を持つ場合（「時刻」「事象」）は漢字のまま。
- **数値と単位**：概数でない数値は必ず数字で書き、単位を付ける。「かなり速い」ではなく「約3倍速い」「200msから80msに短縮」と書く。

## 読点

- 読点は「息継ぎ」ではなく「構造の区切り」として打つ。打つ位置は、長い主語の直後、接続詞の直後、係り受けの誤読を防ぐ位置、の三つに限る。
- 一文に読点が3つ以上必要になったら、読点を増やすのではなく文を分割する。

## 段落と接続

- 一段落は一つの主張とその根拠で構成する。主張が変わったら段落を変える。
- 接続詞は論理関係が変わるときだけ使う。「そして」「また」で同方向の文を繋ぎ続けない。順接の接続詞は多くの場合削除できる。
- 「しかし」「ただし」「一方」など逆接・対比の接続詞は削らない。論理の転換点は明示する。

## 具体性の担保

- 抽象的な主張には、コード・数値・実際のエラーメッセージ・具体的な操作手順のいずれかを一つ添える。添えられない主張は、根拠が薄い可能性を疑い、主張ごと見直す。
- 指示語（「これ」「それ」「この方法」）が2文以上前を指す場合は、指示語をやめて名詞で言い直す。
- 読者の環境依存の記述（バージョン、OS、設定）は、前提として冒頭付近に明示する。

## 推敲の手順

草稿を書き上げたら、次の順で点検する。

1. **音読テスト**：一文ずつ読み、息が続かない文・二回読まないと意味が取れない文を分割または語順修正する。
2. **置換テスト**：「冗長表現の削除」の一覧を検索し、機械的に置換する。
3. **揺れテスト**：主要な用語を列挙し、文書内での表記が一意かを検索で確認する。
4. **指示語テスト**：「これ」「それ」「この」を検索し、参照先が直前の文にあるかを確認する。
5. **接続詞テスト**：「そして」「また」「基本的に」を検索し、削除して意味が変わらないものを削る。

この点検を終えた文章に対してのみ、緩急の設計（`../cognitive-rhythm-writing/SKILL.md`）を適用する。順序を逆にすると、リズムのために残した文が構造の乱れを隠す。
