# worktree-parallel

独立した複数の開発タスクを、**git worktree で隔離したサブエージェント**に並列実行させ、最後にマージバックまで導くためのプレイブック。worktree の作成・隔離は Claude Code の組み込み機能に任せ、このSkillは**手順を束ねるだけ**。独自スクリプトや手動 worktree 管理は書かない。

## 何をするSkillか / いつ使うか

- **使う**：互いに独立した複数の開発タスクを同時に進めたいとき（例：機能Aの実装・機能Bの実装・リファクタC を並行）。
- **使わない（重要）**：タスク間に依存があるとき（Aの結果をBが使う、同じファイルを両方が編集する等）は**並列に向かない**。この場合はサブエージェントを1つずつ**順次実行**するか、タスクを切り直す。判断がつかなければユーザーに確認する。

## 前提ルール

- worktree は必ず `.claude/worktrees/<名前>/` 配下（組み込みのデフォルト置き場）に作る。ここ以外には作らない。
- `.gitignore` に `.claude/worktrees/` が無いと、worktree のファイルが本体側で未追跡ファイルとして見えて事故る。**これを手順の最初のステップで必ず確認・追加する。**

## 手順

1. **タスクの棚卸しと独立性の確認**：各タスクが本当に独立しているかを確認する。共有ファイル・DB・ポートを取り合わないか、同じファイルを複数タスクが編集しないかを見る。依存があればタスクを分割し直すか、ユーザーに並列でよいか確認する。
2. **`.gitignore` の確認**：`.claude/worktrees/` が `.gitignore` にあるか確認し、無ければ追加する。
3. **サブエージェントの並列起動**：各タスクにつき Agent ツールを `isolation: worktree` で1つ起動する。**独立タスクは1つのメッセージで複数の Agent を同時に呼び、並列実行させる**（依存があるものだけ順次）。各サブエージェントには次を必ず指示する：
   - 自分の worktree 内でだけ作業すること。
   - 完了したら変更を**ブランチにコミットする**こと（コミットしないと worktree が自動削除されて成果が消える場合がある）。
   - 完了報告に、成果・変更したファイル・注意点・ブランチ名を含めること。
4. **成果の集約と要約**：各サブエージェントの完了報告を集める。**サブエージェントの出力はユーザーには折りたたまれて見えない前提**で、各エージェントが何をやったか・変更点・注意点を、自分の言葉でわかりやすく要約してユーザーに伝える。
5. **マージバック**：`git worktree list` で状態を確認 → 各ブランチをレビュー → `git merge`（または PR 作成）で本体に取り込む → 取り込み済みのものは `git worktree remove` で片付ける。

## この案が破綻する条件

- worktree は**ファイル編集を隔離するだけ**で、マージ時に同じ行を触っていればコンフリクトは普通に起きる。並列が効くのは「触るファイルが被らないタスク」だけ。被る前提なら並列にしない。
- 並列サブエージェントは計算・トークンを大量に消費する。タスク数が多いほど重くなる。数を絞る。
- 各 worktree は依存（node_modules 等）を共有しない。ビルドが要るタスクは、セットアップコストが worktree の数だけかかる。

## やらないこと

- 手動の `git worktree add` を主手段にしない。組み込みの `isolation: worktree` を使う。
- worktree を `.claude/worktrees/` の外に作らない。
- 独自の worktree 管理スクリプトを書かない（車輪の再発明をしない）。
