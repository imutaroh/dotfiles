# Progressive Learning — サンプルセッション

実際のセッション（Polaris の Go Ingestion / Google広告 の取り込みを抽象から学んだケース）を再構成したサンプル。各 Step が実運用でどう動くかの参考にする。

---

## セッションの初期条件

- 学習対象：Polaris（モノレポ）の `ingestion/unified-api/` 内の Google広告 取り込み
- ユーザー：Go学習中の新卒エンジニア
- 既存資料：assistant が事前に作った要約 HTML がある

ユーザーは要約 HTML の「ひとこと」を読んでも単語が分からず "呪文" 状態。「もっと噛み砕いてほしい」と要望。

---

## Step 0: ゴール文（呪文）の確定

assistant は学習対象の「ひとこと」を確定する。今回はユーザーが指定してきた。

```
[ゴール文（呪文）]
Cloud Scheduler などの外部スケジューラから HTTP で叩かれた unified-api が、
/api/v2/google-ads/* のハンドラを起点に
① MCC配下の対象アカウントを列挙
→ ② アカウント毎に GAQL（Google Ads クエリ言語）で API を叩く
→ ③ BQ 行型に変換
→ ④ BigQuery に Replace か Append で書き込む
という ETL を回している。
「ジョブ」ディレクトリ（internal/jobs/）には Google Ads は無く、
API ハンドラがバッチ実体になっている点に注意。
```

assistant の最初の動き：
- ひとことを「やさしい日本語訳」して提示
- 出てくる新出単語を全部抽出

---

## Step 1: 呪文の解体と進捗バー作成

呪文から新出単語を抽出：

| 単語 | 一言の意味 |
|---|---|
| Cloud Scheduler | クラウドの目覚まし |
| HTTPリクエスト | URLを叩くこと |
| unified-api | データ取り込み専門のお遣いやさん |
| ハンドラ | URLごとの受付係 |
| MCC | Google広告の親アカウント |
| GAQL | Google広告に投げる注文書 |
| BQ行型 | BigQueryの1行分のフォーマット |
| Replace / Append | 上書き vs 追記 |
| ETL | Extract → Transform → Load |
| バッチ実体 | 定期実行される処理の本体 |

初期進捗バー：

```
⬜ Cloud Scheduler
⬜ HTTPリクエスト
⬜ unified-api
⬜ ハンドラ
⬜ MCC
⬜ GAQL
⬜ BQ行型
⬜ Replace / Append
⬜ ETL
⬜ バッチ実体
```

## Step 2: 各概念を「ひとこと → 例え話 → 実物」の3段で

### ターン例：Cloud Scheduler

```markdown
## Cloud Scheduler ってなに？

ひとことで言うと：クラウドの上にある「目覚まし時計」

それだけ。本当にそれだけ。

### 身近なたとえ

朝7時にアラームをセットした目覚まし時計を想像してください。

- 7時になる → ピピピッ！と鳴る
- 鳴ったあと、起きるか二度寝するかは持ち主の自由

Cloud Scheduler も同じノリです。

- 毎朝7時 → 指定された場所に電話をかける
- 電話を受けた相手が何をするかは、相手のサーバー次第

### 「電話をかける」ってどういうこと？

Cloud Scheduler の場合、「電話」の正体は HTTPリクエスト（URLを叩くこと）です。

つまり：

> Cloud Scheduler = 「指定した時刻に、指定したURLを叩いてくれる人」

それ以上でも以下でもないです。

---

次どこ掘ります？選んでください。
[AskUserQuestion で選択肢提示]

進捗：
✅ Cloud Scheduler
⬜ HTTPリクエスト
⬜ ...
```

### ターン例：ハンドラ

```markdown
## ハンドラ ってなに？

ひとことで言うと：URLごとの "担当受付係"

### ホテルのフロントで例える

大きなホテルに行くと、フロントの中にカウンターが何個か分かれてますよね。

[ホテルのフロント]
 ├─ 「宿泊」カウンター     ← チェックイン担当の受付さん
 ├─ 「レストラン」カウンター ← レストラン予約担当の受付さん
 └─ 「コンシェルジュ」      ← 観光案内担当の受付さん

unified-api というホテルにも複数のカウンターがある。
そのカウンターにいる「受付係」がハンドラ。

### 仕事は何？

ハンドラ自身は実はそんなに難しいことしてない。仕事を振り分けるだけ。

1. 注文書を読む（JSON チェック）
2. 「tenant って書いてある？」
3. 奥の本当に仕事する人（service）を呼ぶ
4. 仕事終わったら「終わりました」と返事
```

このターンではコードは出さない。例え話とロールの説明だけ。
ユーザーが納得してから、次のターンか「実物見たい」リクエストでコードを出す。

## Step 3: 道草対応の例

### ユーザーの寄り道質問

> 「これさ、Goの三層構造のhandlerの層のことを指していますか？」

これは本筋（次は MCC の説明）からは外れた質問。でも理解の深まりに直結する重要な気付き。

### assistant の対応

```markdown
大正解！！ まさにそれです。

CLAUDE.md にもこう書いてあります：
[3層構造の引用]

いむたろ様、いいとこ繋いだ。これ気付けると一気に視界開けます。

[3層をホテルで例えて改めて整理：handler/service/repository]

---

次、本筋の続きの ② MCC配下の対象アカウントを列挙 いきます？

進捗：
✅ ハンドラ（しかも 3層構造の handler 層そのもの！）
⬜ MCC配下のアカウント列挙　← 次これ
⬜ GAQL
...
```

ポイント：寄り道質問にしっかり答えてから、進捗バーで本筋復帰。

### もう1つの寄り道例：「個人アプリだとhandlerは1ファイルだった」

ユーザーが「個人開発したアプリだと handler が1ファイルだった、これとの違いは？」と聞いてきた。

assistant：

```markdown
[ファイル分けは "整理整頓" の問題、設計の本質は3層分割]
[Goでは同じディレクトリのファイルは同じパッケージ扱い]
[main.go と handler の役割の違い → ホテルの総支配人と受付係]

---

次、進捗の続きの MCC配下のアカウント列挙 いきます？
```

## Step 4: ユーザーのアウトプットへの応答

ユーザーが自分の言葉で再現してきた例：

> 「使い捨ての関数にそれをおくのね じゃあ、そのあとにその配列かな？スライスか、それをフィルターにかけて、しぼりこまれた状態のものを取得って感じか！」

assistant の対応：

```markdown
そう！パーフェクト！自分の言葉でアウトプットできてるの最高。

軽く補足だけ：

[流れを言葉で再現]
[「取得」が2種類あった話の整理]
[配列とスライスのおまけ豆知識]

---

進む準備OK？次GAQLいくよ！
```

ポイント：
1. 「パーフェクト！」で肯定から入る
2. 補足は "軽く"。完全に1から説明し直そうとしない
3. ユーザーが触れなかったが大事なポイントは "豆知識" として軽く

## Step 5: 既存知識との接続を促す

ユーザーが自身の lunch-bot というアプリと Polaris を比較してきた：

> handler が薄くて、service の中で直接 Slack に投稿して完結。Result 返さない。
> なんで僕のアプリでは必要なくて、今回は handler に返してるんだろう？

assistant の対応：

```markdown
ナイス比較！ここに気付けるのレベル高い。
"呼び出し元が何を欲しがるかで設計が決まる"、というのが本質。

[Polaris は HTTPサーバー → 詳細レスポンス必要]
[lunch-bot は CLI → exit code だけで十分]

[レストラン vs 自販機 で例える]

[3つの観点で違いを整理]

---

いむたろ様の lunch-bot 設計は "過剰設計を避けた" 良い設計。
"自販機型の設計" には、ちゃんと意味があってその形になってます。

進む？それともまだ気になるとこある？
```

ポイント：本人の既存資産（lunch-bot）を活かして、本人の理解を深める。

## Step 6: 次の項目はユーザーが選ばせる

毎ターン末尾で AskUserQuestion を使う。

### 例

```
Q：次、どこを掘る？

A. ETL（取る → 整える → 入れる）
B. MCC と子アカウント
C. ハンドラ（URL受付係）
D. バッチとジョブ
```

### ユーザーが「順番通り」を選ぶこともある

```
Q：次、どれを掛り下げる？

A. 順番通りに言葉で理解しよう！！  ← ユーザーがこれを選択
B. （個別項目）
C. （個別項目）
```

このときは呪文の前から順に拾っていく。

## Step 7: 節目のまとめ

ユーザーが「お昼休憩」と言ったタイミングで、戻ってきた時用のまとめを作る。

```markdown
# 🍙 お昼休憩前まとめ（戻ってきたら読むやつ）

## Polaris の Google広告 Ingestion を一言で
> （現時点の理解を1段落で）

## ストーリー（朝3時の出来事として）
（時系列で起動からBQ書き込みまで）

## 主要登場人物
（役割テーブル）

## 単語辞書（覚えた呪文一覧）
（学んだ単語の一行説明）

## 大事な気付き（いむたろ様の閃き集）
- 💡「全部Appendのほうがいいんじゃ？」→ イミュータブル思想に到達
- 💡「interface かな？」→ ストラテジーパターン発見
- 💡 ...

## 進捗バー（更新版）
...

## 戻ってきたらやること（おすすめ順）
1. このまとめを読み返す（5分で復習）
2. 呪文を読み直す
3. 実物の BigQuery を覗く
```

## Step 8: ゴールの呪文を読み直す（フィナーレ）

全項目完了したタイミングで、最初の呪文をそのまま再掲する。

```markdown
🎯 「ひとこと」の全用語、完全制覇しました！

最初に出した「ひとこと」もう一度貼るので、読んでみてください：

> Cloud Scheduler などの外部スケジューラから HTTP で叩かれた unified-api が、
> /api/v2/google-ads/* のハンドラを起点に
> ① MCC配下の対象アカウントを列挙
> → ② アカウント毎に GAQL（Google Ads クエリ言語）で API を叩く

> という ETL を回している。
> ...

どうですか？最初は呪文だった文章、もう全部わかるはず！
```

これがフィナーレ。ユーザーが「最初は読めなかった文が読めてる」を体感する瞬間。

## 卒業メッセージのテンプレ

セッションが終わるタイミングで、本人の学習姿勢を具体的に褒める：

```markdown
いむたろ様、今日の学習スタイル、めちゃくちゃ良かったです：

- 自分の言葉でアウトプットする
  （「以前と比較してるの？」「全部Appendのほうが？」「interfaceかな？」）

- 直感に従って疑問を出す
  （「なぜGoogleだけ別？」「APIないやつはどうする？」）

- 既知の知識と接続する
  （「3層構造のhandlerと同じだ」「自分のlunch-botと比較」）

新卒でこの動き方できる人、本当に少ないです。
```

具体例を挙げて褒めるのが大事。「すごいですね」より「○○の場面で○○できた」と具体的に。

## このサンプルから抽出できる運用上のヒント

1. **呪文の選び方**：抽象的すぎず、新出単語が10〜20個程度ある "ちょうど良いサイズ" を選ぶ。short すぎると物足りないし、long すぎると消耗する。

2. **道草の歓迎度合い**：ユーザーが寄り道質問をしたら、本筋の進度より深堀りを優先する。寄り道で気付くことの方が本筋より価値あることが多い。

3. **コードを出すタイミング**：呪文の主要単語が全部「ひとこと＋例え話」で説明できた後、ユーザーが「実物見たい」と言ったら出す。最初の半分は完全に概念オンリーで進める。

4. **進捗バーの心理効果**：「あと3個」というゴール感が学習の最後まで続く動機になる。毎ターン出すのは必須。

5. **節目のまとめは "戻る場所" を作る効果**：長いセッションでは、まとめが「ここまでで何を覚えたか」を確認する場所になる。次に学び直す時の入り口にもなる。
