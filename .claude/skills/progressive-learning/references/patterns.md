# Progressive Learning — Patterns & Cheat Sheet

このスキルで繰り返し使う「例え話」「フレーズ」「フォーマット」の引き出し集。SKILL.md の各ステップで困ったら、ここから選んで使う。

## 例え話の引き出し（ドメイン別）

技術概念を身近な比喩に置き換える。ユーザーの生活実感に届くものを選ぶ。

### Webサーバー / API / ルーティング系

| 概念 | 例え |
|---|---|
| サーバー | お店、ホテル、レストラン |
| HTTPリクエスト | 「水ください」と店員に頼む注文 |
| URL | お店の住所、テーブル番号 |
| ハンドラ | URLごとの担当受付係（フロントの各カウンター） |
| ルーター | 電話交換機 + 電話帳 |
| GETメソッド | メニュー見せて |
| POSTメソッド | これ注文します |
| レスポンス | 注文した品＋「お待たせしました」 |

### バッチ / 定期実行系

| 概念 | 例え |
|---|---|
| バッチ | 洗濯機、食洗機（ためて一気に処理） |
| cron | サーバーに住む "世話焼きおじさん" |
| Cloud Scheduler | クラウドの目覚まし時計、複数アラームを管理 |
| やることリスト（crontab） | 目覚まし時計のリスト機能 |
| ジョブ | バッチで動く1個1個の仕事 |

### レイヤード / 役割分担系

| 概念 | 例え |
|---|---|
| Handler層 | 受付係（注文取って厨房に通すだけ） |
| Service層 | 料理長（段取り組む、レシピ知ってる） |
| Repository層 | 食材調達係（外との接点） |
| 共通部品（ETL Runner など） | 共通の調理台、電動ドリル本体 |
| 各プラットフォーム固有部分 | 替えビット（木材用、金属用…） |

### データ管理系

| 概念 | 例え |
|---|---|
| マスタテーブル | クラスのプロフィール帳（最新だけあれば良い） |
| 時系列テーブル | 毎朝の体重記録ノート（履歴を積む） |
| Replace（上書き） | プロフィール帳の住所欄を書き換え |
| Append（追記） | 体重表に今日の値を追加 |
| 変更履歴（change events） | 「誰がいつ何を変えたか」の業務日誌 |
| データウェアハウス | 巨大な倉庫（行と列でフラットに保管） |
| 永続化 vs メモリ保存 | 紙のノート vs スマホのメモ |

### 認証 / 通信系

| 概念 | 例え |
|---|---|
| gcloud auth login | 朝のIDカード認証（会社のセキュリティゲート） |
| OAuth トークン | 入館証 |
| API認証ヘッダ3点セット | ① 入館証 ② 開発者証 ③ 所属部署タグ |

### スクレイピング系

| 概念 | 例え |
|---|---|
| API（公式窓口あり） | 郵便屋さんに依頼、整理された封筒で届く |
| スクレイピング | 雑誌売り場で目当ての記事を破って持ち帰る |
| ヘッドレスブラウザ | 自動操縦のブラウザ |
| サイト改修で壊れる | 売り場のレイアウト変更で迷子 |

### 設計パターン系

| 概念 | 例え |
|---|---|
| interface | 「これができる人ならOK」という契約書 |
| 関数を引数で渡す | 「やり方」を1個だけ渡す |
| ストラテジーパターン | 替えビットを差し込む口 |
| ファイル分け | 整理整頓（規模が大きいときの) |
| パッケージ分け | 役割そのものの分離 |

## 進捗バーのフォーマット

### 標準形

```
✅ 学習済みの概念A
✅ 学習済みの概念B
⬜ 次に学ぶ概念C　← 次これ
⬜ 未習の概念D
⬜ 未習の概念E
```

### 道草吸収版

```
✅ 学習済みの概念A
✅ 学習済みの概念B
✅ 道草で学んだ概念X（おまけ）
⬜ 本筋の次の概念C　← 次これ
```

### フィナーレ前

```
✅ 全項目
✅ ...
✅ ...
⏸ スキップした項目（また今度）
```

## アウトプット肯定フレーズ集

ユーザーが自分の言葉で再現してきたとき：

- 「ほぼ正解！」
- 「めっちゃ良い理解！」
- 「鋭い！」
- 「パーフェクト！」
- 「自分の言葉でアウトプットできてるの最高」
- 「ここ気付けるのレベル高い」
- 「設計の勘がちゃんと育ってきてる」

軽い微補強の入り方：

- 「軽く補足だけ：」
- 「ただ1点だけ整えさせてください」
- 「半分正解、半分違う」（ただし必ず肯定パートを先に）
- 「いい着眼点なんだけど、実は…」

## 道草対応フレーズ集

寄り道質問を受けたとき：

- 「これも超いい疑問！」
- 「ナイス気付き！」
- 「ここで引っかかれるの大事」
- 「設計者の感覚として正しい」

道草終わりに本筋復帰：

- 「これでスッキリしました？進捗のここに戻ろう」
- 「ここまで吸収できたので、進捗に "おまけ：○○" として追加しときます」
- 「では本筋の次に進みましょう」

## 訂正パターン（否定から入らない）

ダメな入り方：「それは違います」「間違ってます」
良い入り方：

- 「なるほどそう来たね、ただ実は…」
- 「半分正解、半分違う」
- 「いい発想なんだけど、もうちょっと深いところがあって」
- 「その感覚は正しい、ただ現実は…」
- 「鋭い目の付け所、ただ整理させてください」

## ユーザーが「わからん」と言ったときの戻し方

- 一段戻って、より抽象に
- 例え話を別ドメインに変える（ホテル → ファミレス、など）
- 一旦コードを全部捨てて、概念だけで描き直す
- 「ごめん、説明が混乱を招きました」と素直に認める
- 「もう一回ゼロから整理させてください」

## セッション中盤の "まとめ" フォーマット

```markdown
# 🍙 ここまでまとめ（休憩前 / 区切り）

## 一言で
> （学習対象の現時点の理解の言語化）

## ストーリー再構築（時系列で）
（朝3時に / 起動して / handler が... の順）

## 主要登場人物
| 名前 | 役割の一言 |

## 単語辞書
| 単語 | 一言の意味 |

## 大事な気付き（ユーザーの閃き集）
- 💡 「○○じゃね？」→ 実は○○思想と一致
- 💡 ...

## 進捗バー（現状）
```

## フィナーレ（卒業）フォーマット

```markdown
# 🎓 今日のゴール達成

## 朝の目標
（Step 0 の呪文を再掲）

## 学んだ世界地図
（ストーリー or 図）

## いむたろ様が手に入れた感覚
- 概念レベル
- 設計感覚
- 実装の見え方

## 最終進捗バー

## 次に学びたくなったときのおすすめ
- 関連分野A
- 関連分野B

## 最後に1個
（ユーザーの学習姿勢を具体的に褒める。今日の閃き／質問の鋭さに言及する）
```

## 落とし穴チェックリスト

セッション中に確認する：

- [ ] 進捗バーをこのターンの末尾に出したか？
- [ ] ユーザーが選ぶ前に勝手に次に進んでいないか？
- [ ] コードを早出ししていないか？（呪文の各単語が概念で理解できてから）
- [ ] 1ターンに詰め込みすぎていないか？
- [ ] 否定形で訂正していないか？
- [ ] 例え話を1つは入れたか？
