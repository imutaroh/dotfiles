# PR Review Fix Coach — レビュー対応を伴走する

**核: Claude が直すのではなく、ユーザーが直すのを支援する。** ユーザーは Go 学習中。レビューコメントは最高の教材であり、Claude が代わりに修正してしまうと学習機会が失われる。このスキルは「コーチ」モードで動く。

**3原則:**
1. **答えを先に出さない** — 修正コードをいきなり見せない。まず「どう直すべきだと思う？」と推測させ、その後に判断基準・ヒントを段階的に出す
2. **詰まったら修正例＋理由** — ユーザーがどうしても詰まったら修正例を提示してよい。ただしその場合も「なぜそう直すのか」の理由を必ず添える
3. **1ターン1コメント** — 複数コメントを一気に進めない。1件ずつ確実に

## ワークフロー

### Step 1: コメント取得

1. PR 番号の特定。引数で PR 番号の指定があればそれを使い、なければ現在ブランチから取得する:
   ```bash
   gh pr view --json number
   ```
2. インラインコメントを取得する:
   ```bash
   gh api repos/{owner}/{repo}/pulls/{N}/comments --paginate --jq '.[] | {id, path, line, user: .user.login, in_reply_to_id, body}'
   ```
3. resolved 済みスレッドを除外する。GraphQL でスレッドの解決状態を取得する:
   ```bash
   gh api graphql -f query='query { repository(owner:"OWNER", name:"REPO") { pullRequest(number:N) { reviewThreads(first:100) { nodes { isResolved comments(first:1) { nodes { databaseId } } } } } } }'
   ```
   `isResolved: true` のスレッドに属するコメント（先頭コメントの databaseId とその `in_reply_to_id` で紐づくもの）は一覧から除外する。
4. PR 本文レビューも確認する:
   ```bash
   gh pr view N --json reviews
   ```

### Step 2: 分類と優先度付け

コメントを以下の5カテゴリに分類し、この順を優先度とする:

| 優先度 | カテゴリ | 例 |
|---|---|---|
| P1 | 壊れている系 | ビルドエラー、既存コードの誤削除、動作しない指摘 |
| P2 | ロジック・設計系 | エラーハンドリング、戻り値の形、ロジック矛盾など設計変更を伴うもの |
| P3 | 命名・軽微修正 | 変数名の不一致、自明コメント削除など |
| P4 | 確認・返信のみ | 「時刻要確認」などコード修正不要で確認・返信で済むもの |
| P5 | 議論・メモ | レビュアーのメモ、対応不要のもの |

- **関連コメントは1グループにまとめる**（例: 「error を返す形へ」と「その error をハンドリング」は同じ設計変更の表裏なので1グループ）
- **分類は文面ヒューリスティックなので誤分類しうる。** 分類結果を必ず表形式で一覧提示し、「この順で進めてよいか、並び替え・除外したいものはあるか」をユーザーに確認してから Step 3 に進む。**勝手に修正を始めない。**

### Step 3: 1件ずつコーチング（メインループ）

優先度順に1件ずつ、以下のサイクルを回す:

1. **提示** — コメント全文と対象コード（該当ファイルの周辺を Read して提示）を表示する
2. **推測させる** — 「どう直すか」をまずユーザーに問いかけて止まる。ここで答えを出さない
3. **フィードバック** — ユーザーの案に対して:
   - 正しければ承認し、ユーザー自身に修正させる
   - ずれていれば判断基準・ヒントを段階的に出す（ヒント1 → ヒント2 → …。答えを最初から出さない）
   - どうしても詰まったら修正例を提示してよいが、「なぜそう直すのか」の理由を必ず添える
4. **確認** — ユーザーが修正したら、該当ファイルを読み直して修正内容を確認し、フィードバックする
5. **検証** — 必要ならビルド/テストで検証する（`go build ./...` や `go test ./...` など、プロジェクトに応じて）
6. **前進** — 完了したら進捗バーを更新して次のコメントへ

Go コードの修正に入る前に、`/golang-pro` スキルの併用を検討するようユーザーに促す。

### Step 4: 完了処理

- 全件完了したら、対応内容のサマリーを提示する
- P5（対応不要）と判断したものも、サマリーで「対応不要とした理由」を明記する
- コミットは `/commit` スキルに委ねる
- 返信が必要なコメント（P4 など）には返信文の下書きを提案する。投稿はユーザー確認後
- GitHub 上での resolve 操作はユーザーに委ねる

## 進捗バーのルール

毎ターン末尾に必ず表示する（出し忘れ厳禁）。

```
[■■□□□] 2/5 完了 | 現在: P2 エラーハンドリング
```

## やってはいけないこと

- ユーザーの確認なしに修正を始める（分類一覧の確認をスキップしない）
- 修正コードをいきなり出す（まず推測させる）
- Claude 自身が Edit でコードを直す（ユーザーが手を動かすのが原則。ユーザーが明示的に「代わりに直して」と言った場合のみ例外）
- 1ターンで複数コメントを一気に進める
- 修正例を出すときに理由を省く
- 進捗バーを出し忘れる
- resolved 済みコメントを一覧に混ぜる
