# 15章チェック観点リスト

The Art of Readable Code (O'Reilly 2012) の各章を、コードに当てるための「匂い (smell) → 改善手法」リスト。
各章の詳細は原著PDF（`/Users/imutaakihiro/repos/androots/the-art-of-readable-code.pdf`）にある。ここはスキャン用の凝縮版。

---

## Part I: 表面的な改善

### ch01 — Code Should Be Easy to Understand
**Key Idea**: 「他の人が最短時間で理解できる」ことを最優先する（TTU: Time To Understand）。短さ < 理解しやすさ。

**匂い**:
- 短いが意図が読めない（`x`, `tmp`, `data`）
- 1行に詰め込みすぎ
- 「なぜそうするか (why)」がコードからは絶対に分からない

**改善手法**:
- 「これは何？」「なぜそれ？」が読み手に伝わるよう、命名・分割・コメントの 3 つを再検討する
- TTU を縮める方向に動かす

---

### ch02 — Packing Information into Names
**Key Idea**: 名前に情報を詰め込め。`Get` の代わりに `Fetch` / `Download` / `Compute` のような **具体的な動詞**。

**匂い**:
- 動詞が汎用（`Get`, `Do`, `Handle`, `Process`）
- `tmp`, `retval`, `i`, `data` のスコープが広い
- 単位が名前にない（`delay` → 秒？ミリ秒？）
- 状態・属性のエンコードがない（`raw_password` / `hashed_password` の区別がない）

**改善手法**:
- 動詞を「目的」を語る単語に差し替え
- 単位サフィックス: `delay_seconds`, `size_bytes`, `latency_ms`
- 状態サフィックス: `_raw`, `_escaped`, `_utf8`
- スコープが短ければ `i` で OK、長いなら `userIndex` などに

---

### ch03 — Names That Can't Be Misconstrued
**Key Idea**: 「誤解されない」名前 > 「良い」名前。読み手が他の意味に取れないかを検証する。

**匂い**:
- `filter` (除外する？ 残す？)
- `length` / `size` (バイト？ 文字？ 要素数？)
- 範囲の `min` / `max` (含む？ 含まない？)
- `first` / `last` (含む？ 含まない？)
- ブール変数の名前に動詞がない（`disable_ssl` より `use_ssl=false` で読み違える可能性）

**改善手法**:
- 範囲: 包含なら `min_` / `max_`、排他なら `first` / `last` / `begin` / `end`
- ブール: `is_`, `has_`, `can_`, `should_` プレフィックス
- 期待される値: `getUsers()` より `getActiveUsers()` で範囲を明示

---

### ch04 — Aesthetics
**Key Idea**: 一貫した、意味のある書式は「読む速度」を上げる。シルエットを揃え、似たものを似た形で並べる。

**匂い**:
- 並列構造（複数の API メソッドなど）でシルエットが揃っていない
- 縦の列が揃わない（似た代入の =、似た引数の型）
- 一貫した順序がない（章ごとにメソッドの並び順がバラバラ）

**改善手法**:
- 並列構造はシルエットを揃える（引数順・改行位置）
- 「列で見える」化（定数の URL を 1 箇所に集める）
- 関数を「段落」に分け、空行で区切る
- 関連メソッドはブロックでまとめる

---

### ch05 — Knowing What to Comment
**Key Idea**: コメントは「書き手が知っていて読み手が知らないこと」を埋めるためにある。コードの言い直しではない。

**コメントすべきでないこと**:
- 関数名を言い直しただけ (`// ユーザーを取得する` の上に `getUser()`)
- 名前を直せば消えるコメント（松葉杖コメント）

**コメントすべきこと**:
- **Director Commentary**: なぜそう書いたか / 代替案を棄却した理由
- **欠陥**: `TODO`, `HACK`, `FIXME`, `XXX`
- **定数**: なぜその値か (`const MAX_RETRIES = 3 // 3 回で十分、それ以上は外部 API が嫌がる`)
- **ありそうな疑問の先回り**
- **落とし穴の予告**
- **Big Picture**: ファイル先頭 / 関数先頭の俯瞰

**匂い**:
- 名前で語れるのにコメントで言い訳している
- コード変更で陳腐化したコメント

---

### ch06 — Making Comments Precise and Compact
**Key Idea**: 同じ情報をより少ない文字で。

**匂い**:
- 3 行コメントで言えることを 5 行書いている
- `it` / `this` / 「それ」が複数のものを指せる
- 「行を返す」「カウントする」のような曖昧動詞
- 関数シグネチャに謎の `true, false, 0, 4` などの positional 引数

**改善手法**:
- 代名詞を実体に置き換える（`it` → `the cache`）
- 入出力例 1 つで仕様を語る
- 「逆順にループ」より「高い順に表示」（意図を語る）
- 名前付き引数 / コメント引数: `f(/* size */ 100, /* sticky */ true)`
- 5 行の説明を 1 単語に圧縮（`caching layer` / `canonicalize`）

## Part II: ループとロジックを単純化

### ch07 — Making Control Flow Easy to Read
**Key Idea**: 制御フローは「自然な向き」で読めるようにする。

**匂い**:
- 二重否定: `if !disabled`
- `if (CONST == value)` のような Yoda 条件
- ネストが深い `if` のピラミッド
- 早期 return しない関数
- `for` の条件式が複雑

**改善手法**:
- 二重否定を肯定形に: `if !disabled` → `if enabled`
- 比較は「変化する値 < 固定値」順: `if length >= 10`
- 早期 return / continue でネストを潰す（ガード節）
- do/while より while
- goto はほぼ常に避ける

---

### ch08 — Breaking Down Giant Expressions
**Key Idea**: 巨大な式は分解する。**説明変数 (explaining variable)** と **要約変数 (summary variable)** が武器。

**匂い**:
- 1 行に `&&` `||` `?:` が 3 つ以上
- メソッドチェーンが 3 段以上
- 同じ式を 2 回書いている (DRY 違反)
- 二重否定 + ド・モルガン適用前

**改善手法**:
- **説明変数**: 部分式に名前を付けて中間変数に
- **要約変数**: 概念にラベルを付ける（`user_owns_document := request.user.id == document.owner_id`）
- **ド・モルガン**: `!(a && b)` → `!a || !b` で否定を内側へ
- **逆から考える**: `OverlapsWith` を「重ならない条件」の否定で書く
- 巨大ステートメントを Summary Variable + DRY で分解

---

### ch09 — Variables and Readability
**Key Idea**: 変数の数を減らし、残った変数を「軽量」に保つ。

**匂い**:
- 不要な中間変数（1 回しか使わない、複雑にしているだけ）
- スコープが広すぎる変数（関数の冒頭で宣言、最後で使う）
- ループ外で宣言してループ内でしか使わない変数
- 再代入が多い変数（追跡コストが高い）

**改善手法**:
- インラインで戻り値を使う（中間変数を消す）
- 変数のスコープを最小化（使う直前で宣言）
- 一度だけ書き込む変数を増やす（const / final / immutable）
- 巨大関数のメンバ変数 → ローカル変数に降格できないか

## Part III: コードの再編成

### ch10 — Extracting Unrelated Subproblems
**Key Idea**: 汎用コードをプロジェクト固有コードから分離せよ。

**匂い**:
- 1 つの関数の中で「ユーティリティ的処理」と「業務ロジック」が混ざっている
- 似た処理を別の場所で何度も書いている
- 関数の半分が「下準備」（フォーマット、変換、検証）

**改善手法**:
- ユーティリティ関数を切り出す（汎用化）
- ライブラリ／ヘルパー群を厚くして、コアの「あなたのプロジェクト固有」部分を小さく保つ
- ただしやりすぎ注意（一発しか使わない関数を切り出すと逆に追跡コストが上がる）

---

### ch11 — One Task at a Time
**Key Idea**: 1 つの関数 / ブロックは 1 つのタスクだけ行う。

**手順**:
1. その関数がやっているタスクを **全部書き出す**
2. タスクを **段落** に分ける（空行で区切る、コメントで小見出し）
3. 段落のいくつかが「実は別の関数になれる」と気付く

**匂い**:
- 関数内で複数の「フェーズ」が混在（取得 → 変換 → 検証 → 保存 が一塊）
- if 内で「判定 + 副作用 + ログ」が同居

**改善手法**:
- 関数を「タスク」単位で関数に切る
- 切るほどでもないものは空行で段落化

---

### ch12 — Turning Thoughts into Code
**Key Idea**: プログラムを **プレーンな日本語** で説明してから書く。説明に使った言葉が、抽出すべき関数名・概念のヒント。

**手順**:
1. 何をしたいかを口語で書き出す
2. 言葉に出てきた「名詞 / 動詞」を関数や変数に対応させる
3. 説明に出てこない処理 = 余計な処理かもしれない

**匂い**:
- コードを読んでも何をしているのか口で説明できない
- 「条件 A の場合、〜だが、条件 B の場合は〜、ただし条件 C のときは〜」のような口語で書き出すと整理できる対象

---

### ch13 — Writing Less Code
**Key Idea**: 最良のコードは「書かないコード」。

**匂い**:
- 標準ライブラリ / 既存ヘルパーで解決済みの問題を自前実装
- 「いつか必要になるかも」で書いた未使用機能
- 巨大なフレームワーク・ライブラリへの不必要な依存

**改善手法**:
- 要件を切り詰める（本当に必要？）
- 標準ライブラリを再発見する（言語の標準機能を読み直す）
- コードを定期的に読み返して削れるところを探す

---

## Part IV: 選り抜きトピック

### ch14 — Testing and Readability
**Key Idea**: テストコードでも可読性は最優先。テストが読みやすい = 書きやすい = 増える = 設計が良くなる。

**匂い**:
- 巨大な setup（1 つのテストに 30 行の準備）
- 入力データと期待値が散在
- アサート失敗時のメッセージが不親切
- テスト名が `TestFoo1`, `TestBar2` のような無情報
- 1 つのテストが複数のことを検証している

**改善手法**:
- ヘルパー関数で setup を 1 行に
- table-driven test（Go なら特に）
- テスト名は「条件 + 期待」: `TestPayment_WhenInsufficient_ReturnsError`
- 1 テスト = 1 アサーション概念
- Bad input / Edge / Stress に対応する小さなテストを足す
- テストしやすい本番コード = 結合度が低い設計

---

### ch15 — Designing and Implementing "Minute/Hour Counter"
**Key Idea**: 実コードは「最終形」ではなく **進化** する。素朴な実装 → 性能改善 → 抽象化、の流れを章でケーススタディしている。

**応用**:
- いきなり最終形を目指さず、段階的に進化させる
- 性能要件が出てから最適化（早すぎる最適化は ch13 の「書きすぎ」）
- 抽象化は重複を見つけてから

---

## チェックの当て方（実践）

このリストを Step 2 で当てるとき、**順序は Part I → II → III → IV** で進める。
直すコストが低い順で、いむたろ様の負担が少ない。

「全部当てる」より「**最も効くもの 5 件**」を提示する方が、いむたろ様の段階的学習に合う。
