# Readable Code Refactor

## 位置づけ

「The Art of Readable Code」15章の観点で **コードを直す作業スキル**。説明・学習ではなく、Diff を一気出ししていむたろ様が読み採るための道具。

- 学習寄りの掘り下げは扱わない（必要なら `learn` スキルへ）
- レビュー指摘リストではなく **改善後のコード（Diff）** を出す
- 「全件出して、いむたろ様が採否を選ぶ」が基本姿勢

## ワークフロー

### Step 1: 対象を確定

- ファイルパス指定があれば即読む
- 「直近変更したやつ」と言われたら `git status` + `git diff` で対象を絞る
- コード片を貼られたら言語と文脈を確認

範囲がはっきりしないときだけ 1 回だけ聞く（章絞る？全章？ファイル全体？関数だけ？）。それ以外は **聞かずに進める**。

### Step 2: 観点を当てて Diff を作る

`references/checklist.md` を読み、15章の観点を Part I → II → III → IV の順で当てる。
見つけた改善は **すぐ Diff に起こす**。途中で「これでよろしいですか？」を挟まない。

### Step 3: Diff を一気出し

見つけた改善を全部、以下のフォーマットで連続出力する。

```markdown
### [ch番号] [一行タイトル]
\`\`\`diff
- 古い
+ 新しい
\`\`\`
```

- 1 Diff = 1 改善（タスク分離）
- 説明は **章タグだけ**（`ch02`, `ch07` 等）。WHY は書かない（読めば分かるはず）
- 章タグだけで意図が伝わらない時に限り、1 行だけメモを足す

### Step 4: 適用

いむたろ様が「全部適用」「ch07 と ch08 だけ」のように言ったら `Edit` ツールで反映する。

- 1 章 = 1 Edit を目安（いむたろ様が後から diff を追える粒度）
- 適用後は、編集したファイルと行数だけ短く報告

### 末尾: 参考リンク（押し付けない）

提案の出力末尾に 1 行だけ。

```
深掘り: 原著PDF（androots/the-art-of-readable-code.pdf）の該当章を参照
```

「もっと解説して」「なぜこれを直すの？」と聞かれたら、その時に `learn` スキルへ渡す。

## やってはいけない

- 改善提案を「表」で整理して優先度を聞く → 直接 Diff を出す
- WHEN/WHY を解説的に書く → 章タグだけで済ます
- 1 件ずつ確認してから次に進む → 全件まとめて出す
- コメントを増やすリファクタを優先する → ch05 違反、命名で直すのが先
- 他言語比較で説明 → 不要
