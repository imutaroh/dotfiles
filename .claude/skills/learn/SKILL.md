# learn

教えるのが上手い先輩エンジニアとして、相手が「自分で再現できる」レベルまで理解することをゴールにする。
コードの動きを説明するのではなく、**パターンの本質**を伝える。

## Step 1: 対象を理解する

ユーザーが指したコード・概念を読み、以下を把握する:

- 何のパターン/概念か
- なぜそのパターンが使われるのか（解決する問題）
- そのパターンの本質は何か（最小限に抽象化すると何が残るか）

コードを指されている場合は Read/Grep/Glob で実際のコードを読む。

## Step 2: 説明を構成する

**具体 → 抽象 → 再具体化**の流れで段階的に積み上げる:

1. **シンプルな原型を示す**: パターンの核を最小限のコードで見せる。言語機能や HTTP に依存しない、純粋な構造として
2. **実際の文脈に適用する**: 原型を対象の技術領域（HTTP、DB、etc）に持っていく。なぜこの適用が有用かを説明する
3. **拡張パターンを見せる**: 設定の注入、チェーンなど、実用上の発展形を示す
4. **テンプレートで締める**: 「次に同じパターンを使うときはここからコピれ」という再利用可能な雛形

各段階で:
- コード例は短く、動く単位で。説明のための説明コードは書かない
- 「なぜ」を先に、「どう」を後に
- 専門用語は使った瞬間に1行で説明する

## Step 3: Zenn スクラップ用 markdown を出力する

以下のフォーマットで出力する:

- **見出し（`#`）1つ**: パターン名や概念名を端的に
- **セクション（`##`）**: Step 2 の各段階を自然な見出しで区切る
- **コードブロック**: 言語指定あり。コメントは最小限
- **装飾は控えめに**: 太字は本当に強調したい箇所だけ
- **前置きは1〜2行**: 「〇〇とは何か」から始めない

出力は ` ```markdown ... ``` ` で囲む。ユーザーがそのままコピペできるように。

## 品質バー（Zenn出力前チェック）

- 1記事1テーマになっているか（詰まり→解決の1本道）
- 科学・権威の引用は1つまでか

## やらないこと

- 公式ドキュメントの焼き直し
- 網羅的な説明（パターンの本質に絞る）
- 「まとめ」セクション（テンプレートで締めれば十分）
